# TASLP TS-ASR Experiment Configs

This folder defines the recommended experiment matrix for the current
`DynaTaR-Qwen` TASLP submission line.

The main method story is:

1. baseline DynaTaR-Qwen TS-ASR
2. `+ overlap-aware acoustic supervision`
3. `+ target-faithfulness regularization`
4. `+ label-guided router supervision`

The config files are organized into two groups.

For the current full run list, see:

- `confs/taslp/this_round_full_matrix.yaml`

## 1. Main paper ablations on `train-merge`

- `taslp_trainmerge_b0_baseline.yaml`
- `taslp_trainmerge_b1_overlap.yaml`
- `taslp_trainmerge_b2_overlap_tc.yaml`
- `taslp_trainmerge_b3_full.yaml`

These four runs should share the same:

- base Qwen model size
- target-aware encoder size
- optimizer policy
- training split
- decode settings

Only the three proposed supervision branches change across the ablation.

## 2. Matched benchmark experiments

### 2-speaker benchmark

- `taslp_train100_b0_baseline.yaml`
- `taslp_train100_b3_full.yaml`

Use these to report the gain on the matched 2-speaker LibriMix setting.

### 3-speaker benchmark

- `taslp_train3mix_custom_b0_baseline.yaml`
- `taslp_train3mix_custom_b3_full.yaml`

Use these to report the gain on the matched multi-speaker custom setting.

### Single-node full-supervision reference run

- `taslp_train100_full.yaml`

Use this when you want a single-node full-supervision run on `train-100`
without mixing it into the multi-GPU ablation launcher.

## Not included here

The following baselines use different scripts or recipe paths and are not
represented by these TS-ASR configs:

- mixture-only Qwen3-ASR baseline
- cascade TSE -> ASR baseline

## Expected stage usage

Typical usage:

```bash
bash run.sh --config confs/taslp/taslp_trainmerge_b3_full.yaml --stage 4 --stop_stage 9
```

Batch submission helpers:

```bash
bash submit_run_full_matrix.sh
bash submit_run_taslp.sh this_round
bash submit_run_taslp.sh trainmerge_ablation
bash submit_run_taslp.sh benchmark_2spk
bash submit_run_taslp.sh benchmark_3spk
bash submit_run_taslp.sh train100_full
bash submit_run_taslp.sh full_matrix
```

`bash submit_run_full_matrix.sh` is the dedicated one-shot launcher for the
entire current round.

`bash submit_run_taslp.sh full_matrix` remains the equivalent lower-level form.

Override stages when needed:

```bash
RUN_STAGE=7 STOP_STAGE=7 bash submit_run_taslp.sh trainmerge_ablation
RUN_STAGE=8 STOP_STAGE=9 bash submit_run_taslp.sh benchmark_2spk
```

Recommended paper order:

1. run `train-merge` ablations first
2. pick the best full model
3. run matched `train-100` and `train-3mix-custom` baseline/full pairs

## Auxiliary-label expectations

Configs with overlap or router supervision explicitly point to:

- `data/clean/<split>/aux_label_manifest.jsonl`

These files are expected to be generated by `run.sh` stage 4.
The manifest is LMDB-backed:

- runtime only reads `storage: "lmdb"` records
- payloads live in `data/clean/<split>/aux_labels.lmdb`
- `train-merge` builds its own merged `aux_labels.lmdb`

## Eval label policy

`eval_aux_label_manifest` is intentionally left empty in all configs so that:

- training uses auxiliary supervision
- dev `eval_loss` remains comparable to earlier pure-ASR runs

If later analysis needs dev overlap metrics, enable that manifest explicitly in
a copied config.
